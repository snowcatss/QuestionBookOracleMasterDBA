# 企画書（改訂版）: ORACLE MASTER Gold DBA 2019 問題集 Webアプリケーション

> 本書は初版企画書に、ゲーミフィケーション（レベル・コンボ・統計）の計算ロジック確定、集約テーブル導入、採点ロジック、RLS・トリガ、問題データ投入方法を追記・整理した改訂版です。**バイブコーディング時はこの企画書と `supabase_schema.sql` を「唯一の正」として参照してください。**

## 1. プロジェクト概要

- **目的:** ORACLE MASTER Gold DBA 2019 試験対策のための個人・少人数グループ向けWebアプリケーション
- **ターゲットユーザー:** ORACLE MASTER Gold 資格取得を目指す受験者
- **主な特徴:** 実際の試験に近いインターフェース、PC・スマートフォン完全対応のレスポンシブデザイン、派手さを抑えたシンプルで直感的なデザイン（非商用・個人学習用）

## 2. 技術スタック

- **フロントエンド:** Vue.js 3 (Composition API) + TypeScript（SFC構成。CDN版プロトタイプはViteプロジェクトへ移行する）
- **スタイリング:** Tailwind CSS
- **バックエンド / DB:** Supabase (PostgreSQL, Auth, Storage, RLS)
- **ホスティング:** Vercel

## 3. UI/UXデザインコンセプト

- **テーマ:** ミニマリズム（装飾を最小限に）
- **カラーパレット:**
  - 背景: ホワイト＆ライトグレー（`#f8f9fa` など）
  - プライマリー: ブルー系（`blue-600`）
  - アクセント: 正解＝水色（`cyan-500`）／不正解＝赤（`red-500`）／コンボ＝緑（`green-500`）／未出題＝グレー
- **タイポグラフィ:** SQLクエリは必ず等幅フォント（Monospace）＋グレー背景のコードブロックとして描画する。

## 4. ゲーミフィケーション仕様（★確定）

本アプリの学習継続の核。**すべての数値は「DB全体の全問題数（＝親カテゴリ配下の全問題数、以下 `N`）」を基準に算出する。** ユーザーがトップでどのカテゴリを選択中でも、レベル・統計の分母は常に `N` で固定する（＝学習全体の進捗を示す）。

### 4-1. 問題ごとの最新状態（未出題／ミス／ヒット）

各問題はユーザーごとに**排他的に**次の3状態のいずれかを取る。集約テーブル `user_question_status` の `last_result`（最新解答の正誤）で判定する。

| 状態 | 定義 | 表示色 |
|------|------|--------|
| 未出題 | `user_question_status` に行が無い（一度も解答していない） | グレー |
| ミス | `last_result = false`（最後の解答が不正解） | 赤 |
| ヒット | `last_result = true`（最後の解答が正解） | 水色 |

- **必ず「未出題 ＋ ミス ＋ ヒット ＝ N」が成立する**（ユニーク・排他集計）。同じ問題を2回正解しても「ヒット」は+1のまま（延べではない）。
- 一度ヒットした問題を後で間違えると、その問題は「ミス」へ移動する（`last_result` を上書き）。
- 検算例（プロトタイプの数値）: 未出題283 ＋ ミス27 ＋ ヒット276 ＝ 586 ＝ N。

### 4-2. コンボ（連続正解数）

- **定義:** 直近から連続して正解し続けている問題数。`profiles.current_combo` に保持する。
- **カウント対象:** 自由演習・模擬試験の**両モード**でカウントする。
- **リセット条件:** **不正解した時点で 0 にリセット**する。途中中断（×終わる）や画面遷移ではリセットしない（現在値をそのまま維持し、次の解答へ継続する）。
- セッションをまたいでも、不正解しない限りコンボは継続する。

### 4-3. レベルとスタンプシート（合計40スタンプ ＝ レベル最大40）

**スタンプ1個 ＝ レベル1。** 4つの行それぞれ最大10個、点灯したスタンプの総数がそのままレベルになる。

| 行 | %／回数の定義 | スタンプ点灯条件 | 最大 |
|----|--------------|-----------------|------|
| **ヒット** | `ヒット% = ヒット問題数 ÷ N × 100` | 10%ごとに1個（`floor(ヒット% ÷ 10)`） | 10 |
| **コンボ** | `コンボ% = current_combo ÷ N × 100` | 10%ごとに1個（`floor(コンボ% ÷ 10)`） | 10 |
| **模試80%** | 1回の模擬試験で正答率80%以上を達成した**回数** | 1回につき1個 | 10 |
| **模試90%** | 1回の模擬試験で正答率90%以上を達成した**回数** | 1回につき1個 | 10 |

- `レベル = ヒットスタンプ + コンボスタンプ + 模試80スタンプ + 模試90スタンプ`（0〜40）。
- **コンボ・スタンプの重要仕様:** コンボ%は**現在の連続正解数基準**。したがって**不正解でコンボが0に戻ると、コンボ%も0%になり、点灯していたコンボ・スタンプも消灯する**（＝コンボ行のスタンプは増減する）。一方、ヒット・模試のスタンプは累積で減らない。
- 模試80%・模試90%は同一セッションで両方満たすことがある（例: 92%正解なら80%行と90%行の両方に+1）。各行10回で上限（それ以上はカウントしても点灯は10個で頭打ち）。
- 検算例: ヒット47%→4個、コンボ0%→0個、模試80% 0回→0個、模試90% 0回→0個 ＝ レベル4（プロトタイプと一致）。

### 4-4. トップ統計ヘッダーの数値

- 未出題／ミス／ヒットは 4-1 のユニーク・排他集計値をそのまま表示。
- コンボは `profiles.current_combo`（実数）を表示。
- これらは後述の `get_user_stats()` 関数（RPC）で一括取得する。

## 5. 採点ロジック（★確定）

### 5-1. 正誤判定ルール

- **完全一致方式:** ユーザーが選択した選択肢IDの集合が、正解選択肢ID（`choices.is_correct = true`）の集合と**完全一致**したときのみ正解。部分点は無し（ORACLE MASTER本番に準拠）。
- 順序は問わない（集合として比較）。未選択（空選択）は不正解。
- MULTIPLE問題で「あと何個選べます」等の**正解数ヒントはUIに出さない**（設問文に「2つ選択」等が書かれている前提）。

### 5-2. 判定の実行場所

- **正の判定はDB関数 `submit_answer()`（RPC）で行う**（改ざん防止・統計更新の原子性のため）。この関数が正誤判定・履歴記録・集約テーブル更新・コンボ更新を1トランザクションで実施する。
- **自由演習モードでは、解説を即時表示する目的でフロントエンドでも正誤・正解選択肢の突き合わせを行う**（`choices.is_correct` を取得して選択と比較）。ただし記録・スコアの確定値はあくまでDB関数の戻り値を採用する（フロントの判定は表示用）。
- 模擬試験モードは試験中に正解を見せないため、フロント判定は行わず、DB関数の結果のみを用いる。

## 6. 画面別 詳細要件

### ① ログイン画面
- Supabase Auth（メール＋パスワード）。ユーザー名/Eメール入力、パスワード、「ログインを記憶する」、ログインボタン。
- 付加リンク: 新規登録、パスワード再設定、確認メール再送信。

### ② トップ（ダッシュボード）画面
- 上部統計: 未出題／ミス／ヒット／コンボ（`get_user_stats()` の値）。
- レベル表示（`current / 40`）＋スタンプシート展開（4-3 の4行×10スタンプを描画）。
- 前回の演習記録（モード＋日時）。
- モード選択タブ: 自由演習／模擬試験。
- 出題対象フィルタ: 未出題・ミス・ヒット・コンボ（4-1 の状態でフィルタ）＋お気に入り絞り込み・出題順指定。
- 分野（カテゴリ）選択: **親カテゴリ「ORACLE GOLD DBA 2019」の子カテゴリ**をチェックボックスでリスト表示。各カテゴリ横に**保有問題数**（`category_id` 別の `questions` 件数）を表示。
- 下部固定ボタン: 未選択時は非活性、選択時に「学習を開始する（N分野選択中）」で活性化。

### ③ 問題演習画面
- 上部バー: 進行状況（自由演習＝プログレスバー／模擬試験＝残り時間タイマー）、右上「×終わる」。
- 問題メタ: 問題ID、履歴（例: ×—————）、前回出題日時。
- 問題エリア: 設問テキスト、添付画像（`question_images` があればサムネイル→クリックで拡大）、SQLコードブロック強調。
- 選択肢: `question_type` により `SINGLE`＝ラジオボタン／`MULTIPLE`＝チェックボックスを動的レンダリング。
- 自由演習: もどる／すすむ、正解／参考（アコーディオン展開で解説表示）。
- 下部固定バー: もどる／すすむ、メモ、☆お気に入り（`user_bookmarks`）。

### ④ 演習結果画面
- 上部サマリー: 正解数／全体、正答率%。
- 詳細リスト: 連番、正誤（○/×）、問題ID（クリックで該当問題へ）、設問冒頭プレビュー。**このリストは `session_questions`（出題順）と `user_answers` を突き合わせて生成する。**
- 下部固定バー: 「学習履歴へ（もどる）」、「ミスに再挑戦」（当該セッションで `last_result=false` の問題を再出題）。

## 7. データベース設計（Supabase / PostgreSQL）

詳細なDDLは `supabase_schema.sql` を参照。テーブル一覧と初版からの変更点は以下の通り。

| テーブル | 役割 | 初版からの変更 |
|----------|------|----------------|
| `profiles` | ユーザー統計。`current_combo`（状態として保持）等 | 変更なし（`level`/`total_*` は集計から算出、`current_combo` のみ状態保持） |
| `categories` | 階層カテゴリ（親=資格名、子=分野） | **親カテゴリ「ORACLE GOLD DBA 2019」を必須化。分野はすべてその子として登録** |
| `questions` | 問題本体 | 変更なし |
| `question_images` | 問題・解説用画像（複数対応） | 変更なし |
| `choices` | 選択肢（`is_correct`） | 変更なし |
| `exam_sessions` | 演習/試験セッション | **`total_questions` 列を追加**（模試の正答率＝`score ÷ total_questions` の分母。スタンプ判定に使用） |
| `session_questions` | **【新規】** セッションの出題問題と順序 | 新規（結果画面・中断再開・同一模試の再挑戦に使用） |
| `user_answers` | 全解答履歴 | 変更なし |
| `user_bookmarks` | メモ・お気に入り | 変更なし |
| `user_question_status` | **【新規】** 問題ごとの最新状態の集約 | 新規（未出題/ミス/ヒット判定とフィルタを高速化） |

### 集約テーブル `user_question_status`
`(user_id, question_id)` を主キーとし、`last_result`（最新正誤）、`correct_count`、`wrong_count`、`last_answered_at` を保持。トップ統計・フィルタ・履歴表示をこの1枚で高速に解決する。`user_answers`（全履歴）は監査・詳細分析用として併存させる。

## 8. 主要ロジック（DB関数・トリガ）

すべて `supabase_schema.sql` に実装。

1. **`handle_new_user()` トリガ:** `auth.users` への新規登録時に `profiles` 行を自動生成（初回ログイン時の統計取得失敗を防止）。
2. **`submit_answer(question_id, selected_choice_ids, session_id)` 関数（RPC）:** 完全一致で採点し、`user_answers` 追記 → `user_question_status` upsert → `profiles.current_combo` 更新（正解+1／不正解0）を1トランザクションで実施し、正誤（boolean）を返す。
3. **`get_user_stats()` 関数（RPC）:** ログインユーザーの未出題／ミス／ヒット数、コンボ、ヒット%・コンボ%、各スタンプ数、レベル（0〜40）をJSONで返す。トップ画面はこの1回の呼び出しでダッシュボードを描画できる。
4. **`get_category_counts()`（任意）:** 子カテゴリごとの保有問題数を返す（分野選択リスト用）。

## 9. セキュリティ（RLS）

- 全テーブルでRLSを有効化。
- 問題系（`categories` / `questions` / `question_images` / `choices`）は認証済みユーザーが**閲覧のみ**可。投入は管理者（サービスロール）が行う。
- 個人データ（`profiles` / `exam_sessions` / `session_questions` / `user_answers` / `user_bookmarks` / `user_question_status`）は**本人のみ**読み書き可。
- **INSERT時は `WITH CHECK (auth.uid() = user_id)` を明示**し、他人のIDでの行挿入を防止する（初版の `USING` のみでは INSERT が保護されないため修正）。

## 10. 問題データの投入方法（★確定）

数百問を手作業INSERTしないため、**JSONインポートスクリプト**方式を採用する。

1. **画像の先行アップロード:** 問題・解説用画像を Supabase Storage の専用バケット（例: `question-images`）へ先にアップロードし、公開URL（または署名URL）を取得する。
2. **問題JSONの用意:** 1問＝1オブジェクトのJSON配列を用意。カテゴリ名、設問、選択肢（正誤フラグ付き）、解説、`question_type`、画像URL（1で取得したもの）を含める。想定スキーマ例:
   ```json
   [
     {
       "category": "DDL文",
       "content": "次のSQL文で索引を作成しました。...\n```sql\nCREATE INDEX ind_price ON new_products(list_price) INVISIBLE;\n```",
       "question_type": "MULTIPLE",
       "explanation": "INVISIBLE句を指定することで不可視索引を作成できます。...",
       "choices": [
         { "content": "ユーザーからは見えない索引である", "is_correct": false },
         { "content": "不可視索引である", "is_correct": true },
         { "content": "デフォルトでは使用されない索引である", "is_correct": true }
       ],
       "images": [
         { "image_url": "https://.../q_26936_1.png", "image_type": "QUESTION", "display_order": 1 }
       ]
     }
   ]
   ```
3. **インポートスクリプト:** Node.js（`@supabase/supabase-js`、**サービスロールキー**使用）で、カテゴリ名→`category_id` 解決（無ければ子カテゴリを作成）、`questions`→`choices`→`question_images` の順にトランザクション投入する。冪等性のため、再実行時の重複防止キー（例: 外部問題番号）を持たせると安全。
4. スクリプトはローカル実行専用。サービスロールキーは `.env` に置きコミットしない。

## 11. 実装上の注意（バイブコーディング用チェックリスト）

- CDN版Vueプロトタイプは見た目確認用。実装は `npm create vue@latest`（Vite + TS + SFC）で組み直す。
- `.env` に Supabase URL / anon key。`supabase gen types typescript` で型を生成し、以降の実装を型で拘束する。
- 本企画書の第4〜5章（レベル/コンボ/採点ロジック）を `CLAUDE.md` に転記し、AIが仕様からブレないようにする。
- 「全問題数 `N`」は単一資格アプリ前提で「全 `questions` 件数」とする（将来複数資格化する場合は親カテゴリでの絞り込みに拡張）。
- 模試の正答率は `score ÷ NULLIF(total_questions, 0)` でゼロ除算を回避。

## 12. 著作権に関する注意

本アプリで扱う問題文・選択肢・解説は、自作問題または利用許諾のある教材・公式サンプルを出典とすること。試験本番の問題文をそのまま転載することは、個人・少人数利用であっても著作権上のリスクがあるため行わない。
